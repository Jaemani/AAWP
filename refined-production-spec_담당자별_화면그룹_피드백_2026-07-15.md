## 이 피드백의 한 문장 목표

현재 Spec의 관리 콘솔을 기능 메뉴 모음으로 보여주지 말고, 로그인한 담당자가 자기 책임 업무부터 시작하여 같은 사업의 정책·운영·명부·결재·지급·발행 결과를 이해할 수 있는 역할별 업무 공간으로 재구성해주세요.

첫 화면을 보고 사용자가 다음 세 가지를 바로 알 수 있어야 합니다.

1. 나는 지금 어떤 담당자 자격으로 들어왔는가?
2. 지금 내가 처리해야 하는 사업과 업무는 무엇인가?
3. 내가 처리한 뒤 어느 담당자에게 무엇이 넘어가는가?

3. 화면에서 정책과 운영계획을 같은 사업 문맥으로 합치되, 정책 데이터·명부 데이터·지급 기록·발행 기록의 책임과 권한까지 하나로 합치지 마세요.

5. 사용자가 여러 역할을 가진 경우 메뉴를 모두 한꺼번에 합치지 말고, 현재 업무영역을 선택하거나 전환하게 하세요.
6. 첫 PoC의 주인공은 범용 정책 작성기가 아니라 청년 또는 청소년에게 지역화폐를 지급하는 구체적인 바우처 사업 한 건입니다.

8. 정책 승인, 명부 승인, 지급 실행, 발행 승인, 최종 서명을 서로 다른 판단과 행동으로 유지하세요.
9. 화면 기본 문구에는 영문 코드와 내부 식별자를 노출하지 마세요. 코드와 기술정보는 상세 펼쳐보기에서만 보이게 하세요.

## 회의에 따라 바뀐 이해

### 조사에서 적용으로 전환

공무원 백오피스 화면을 더 수집하는 것이 현재 과제가 아닙니다. 조사에서 확인한 역할 분리, 목록·상세·결재·명부·지급 연결 방식을 우리 백오피스에 적용해야 합니다.

### 정책과 운영계획을 한 사업 문맥으로 통합

회의에서 가장 직접적으로 나온 요구는 정책과 운영계획을 따로 분리하지 말고 한 화면에서 보여달라는 것이었습니다. 이를 모든 입력 권한을 한 사람에게 주라는 의미로 해석하지 마세요.

- 정책 담당자는 정책 근거와 조건을 편집하고 명부·지급 상태는 읽습니다.
- 사업·명부 담당자는 승인된 정책을 읽고 회차와 명부를 편집합니다.
- 결재자는 근거와 변경사항을 읽고 승인·보완·반려를 판단합니다.
- 지급 담당자는 승인된 정책과 확정 명부를 읽고 지급을 실행합니다.
- 발행 담당자는 정책을 수정하지 않고 발행 필요성과 발행 준비를 처리합니다.

같은 사업 페이지를 공유하되 현재 역할에 따라 편집할 수 있는 구역과 가능한 행동이 달라져야 합니다.

### 범용 정책 작성기보다 바우처 한 건 우선

첫 PoC에서는 “청년·청소년 지역화폐 지급 바우처” 한 건이 정책 작성부터 지급 결과까지 이어지는 모습을 먼저 완성합니다. 조건 규칙은 재사용할 수 있지만 정책 페이지는 해당 사업에 맞는 고정형 화면이어도 됩니다.

## 현재 화면에서 확인된 핵심 문제

### 1. 로그인과 콘솔 선택

![로그인과 콘솔 선택](audit_2026-07-14/01-login-and-console-choice.png)

현재는 로그인 전에 사용자가 관리 콘솔과 발행사 콘솔을 고르게 되어 있습니다. 사용자는 인증되기 전 자신의 권한을 정확히 알기 어렵습니다. 인증을 먼저 하고, 권한이 하나면 해당 업무영역으로 바로 이동하고, 권한이 여러 개일 때만 업무영역 선택을 보여주세요.

### 2. 정책 목록

![정책 목록](audit_2026-07-14/02-policy-list.png)

현재 정책 목록은 모든 정책 데이터를 분류하는 화면에 가깝고, 정책 담당자가 오늘 무엇을 처리해야 하는지 보여주지 못합니다. 상단에 충전 정책, 상품권 정책, 일반 정책 작성 행동이 동시에 있고, 사업연도·처리기한·보완 여부·다음 담당자가 보이지 않습니다.

### 3. 상품권 정책 작성

![상품권 정책 작성](audit_2026-07-14/03-voucher-policy-form.png)

현재 화면은 정책명, 설명, 대상 조건, 지급 금액까지만 보입니다. 사업 근거, 담당부서, 시행기간, 사용 조건, 재원, 회차, 명부, 결재, 지급 준비로 이어지지 않습니다. 대상 코드가 사람이 읽는 조건보다 강하게 보이고, “저장 및 다음” 이후의 흐름도 알 수 없습니다.

### 4. 역할과 관할 관리

![역할과 관할](audit_2026-07-14/07-role-and-jurisdiction.png)

현재 화면은 역할 코드와 시스템 용어가 중심이고, 역할별 실제 업무 공간이 어떻게 달라지는지 확인하기 어렵습니다. 역할 부여 화면에서만 역할을 나누고 실제 내비게이션은 동일하면 담당자 분리가 되지 않습니다.

### 5. 발행 계획 목록

![발행 계획 목록](audit_2026-07-14/05-issuance-plan-list.png)

발행사 콘솔에 “새 정책 수립”이 남아 있고 관리 콘솔과 같은 정책·유통·정산·거버넌스 메뉴가 보입니다. 정책 담당자와 발행 담당자의 책임이 화면에서 다시 섞입니다.

### 6. 발행 승인

![발행 승인](audit_2026-07-14/06-issuance-plan-approval.png)

발행 승인과 최종 서명이 하나의 버튼으로 묶여 있고, 기술 검증 정보가 정책 목적·대상·금액·지급 일정·근거보다 먼저 보입니다. 발행 승인자와 최종 서명자의 화면과 행동을 분리해야 합니다.

## 담당자별 화면그룹

다음 화면그룹은 조직도를 확정하는 것이 아니라 제품에서 책임과 화면을 구분하기 위한 작업 기준입니다. 실제 조직에서 한 사람이 여러 역할을 겸할 수 있지만, 한 화면에서 서로 다른 책임을 동시에 수행하게 만들지는 마세요.

### 화면그룹 A. 정책 담당자

이 담당자는 사업의 행정 근거, 대상 조건, 지급액, 사용 조건, 기간과 재원 초안을 만들고 결재를 상신합니다.

시작 화면:

- 내 정책 업무

기본 메뉴:

- 내 정책 업무
- 바우처 사업
- 결재 진행
- 승인된 조건 기준

주요 화면:

- 내 처리 필요 정책 목록
- 바우처 사업 업무 상세
- 바우처 정책 작성·보완
- 정책 변경 비교
- 내가 올린 결재 진행

할 수 있는 행동:

- 새 바우처 사업 초안 만들기
- 임시저장
- 검토하기
- 결재 상신
- 보완 요청에 따라 수정
- 반려 사유 확인 후 개정안 만들기

할 수 없는 행동:

- 자신의 정책을 최종 승인
- 명부를 최종 승인
- 지급 실행
- 발행 승인 또는 최종 서명
- 시민 잔액 직접 수정

### 화면그룹 B. 사업·명부 담당자

이 담당자는 승인된 정책을 기준으로 사업연도·회차·기준일을 관리하고, 대상 파일을 반입·검증·보완하여 실제 지급 명부와 총액을 만듭니다.

시작 화면:

- 내 명부·지급 준비 업무

기본 메뉴:

- 내 업무
- 대상·명부
- 지급 준비
- 보완·오류

주요 화면:

- 명부 작업 목록
- 바우처 사업 업무 상세의 대상·명부 구역
- 파일 업로드와 검증
- 오류 수정·재업로드
- 명부 버전 비교
- 명부 확정 상신

할 수 있는 행동:

- 사업 회차와 기준일 입력
- 원본 명부 파일 업로드
- 정상·주의·오류·제외 건 확인
- 오류 목록 다운로드
- 수정본 재업로드
- 제외 사유 기록
- 확정 대상 수와 총액 확인
- 명부 결재 상신

할 수 없는 행동:

- 승인된 정책의 대상 조건이나 지급액 변경
- 자신이 만든 명부를 최종 승인
- 지급 실행
- 발행 계획 승인

### 화면그룹 C. 정책·명부 결재자

이 담당자는 작성 화면을 대신 조작하지 않고, 제출된 정책 또는 명부의 근거와 변경사항을 보고 보완·반려·승인을 판단합니다.

시작 화면:

- 내 결재함

기본 메뉴:

- 승인 대기
- 내가 처리한 결재
- 보완 후 재상신
- 완료

주요 화면:

- 역할에 맞게 필터링된 결재함
- 정책 결재 상세
- 명부 결재 상세
- 변경 비교
- 결재 이력

결재 상세에서 먼저 보여줄 내용:

- 어떤 사업의 어떤 결재인지
- 사업연도, 담당부서, 기안자, 처리기한
- 정책의 목적과 행정·법적 근거
- 첨부파일과 문서번호
- 이전 버전에서 달라진 내용
- 대상 수와 지급 총액
- 명부 결재인 경우 명부 버전, 정상·오류·제외 건수
- 검증 경고와 미해결 항목

할 수 있는 행동:

- 보완 요청
- 반려
- 승인

할 수 없는 행동:

- 결재 상세 안에서 원안의 정책 조건이나 명부 행 직접 수정
- 승인과 동시에 지급 실행
- 승인과 동시에 발행 최종 서명

### 화면그룹 D. 지급 실행 담당자

이 담당자는 승인된 정책, 확정 명부, 재원, 유통 단위와 발행 준비상태를 확인하고 지급을 시작하거나 진행 결과를 관리합니다.

시작 화면:

- 지급 실행 대기

기본 메뉴:

- 실행 대기
- 처리 중
- 일부 실패
- 완료 내역

주요 화면:

- 지급 업무 목록
- 지급 준비 상세
- 지급 실행 확인
- 진행 현황
- 실패 건 상세와 재처리
- 지급 결과

할 수 있는 행동:

- 지급 차단 사유 확인
- 지급 대상 수·총액·명부 버전 재확인
- 재인증 후 지급 시작
- 실패 대상만 재처리
- 지급 결과와 감사 기록 확인

할 수 없는 행동:

- 정책 내용 수정
- 확정 명부 행 임의 수정
- 발행 승인
- 지급 완료 수치를 직접 덮어쓰기

### 화면그룹 E. 발행 계획 담당자

이 담당자는 관리 콘솔과 분리된 발행사 콘솔에서 승인된 정책과 지급 수요를 읽고 신규 발행 필요성을 검토하고 발행 계획을 작성합니다.

시작 화면:

- 발행 필요 정책

기본 메뉴:

- 발행 필요 정책
- 발행 계획
- 승인 진행
- 발행 내역
- 준비자산

주요 화면:

- 신규 발행 검토 목록
- 발행 계획 작성
- 발행 계획 결재 진행
- 발행 준비상태

할 수 없는 행동:

- 새 정책 수립
- 정책 조건 수정
- 정책 결재
- 명부 수정
- 최종 서명자 권한으로 서명

### 화면그룹 F. 발행 승인자·최종 서명자

발행 승인과 최종 서명은 한 버튼으로 합치지 않습니다.

발행 승인자:

- 정책 근거, 대상 수, 지급 총액, 시행일, 발행 필요량과 준비자산 상태를 검토
- 승인 또는 반려
- 발행 계획 본문 직접 수정 금지

최종 서명자:

- 승인 완료된 발행 계획만 조회
- 발행량, 대상 유통 단위, 승인 이력과 준비상태를 최종 확인
- 재인증과 다중서명 절차 수행
- 서명 결과 확인

두 역할을 한 사람이 겸하는 경우에도 화면 상태와 행동은 두 단계로 분리하고, 승인 직후 자동 서명하지 마세요.

### 화면그룹 G. 정산 담당자

기존 정산 화면은 보존하되 정책 담당자 메뉴에서 분리합니다.

기본 메뉴:

- 정산 현황
- 가맹점별 명세
- 예외·취소
- 대사
- 상환

첫 바우처 시연에서는 핵심 흐름 뒤에 두고, 정책 담당자의 기본 메뉴에 노출하지 마세요.

### 화면그룹 H. 감사·리스크 담당자

감사 담당자는 기본적으로 읽기 전용입니다.

기본 메뉴:

- 감사 로그
- 결재·변경 이력
- 정합성 리포트
- 거래 추적
- 처분 이력

정책, 명부, 지급, 발행 기록을 수정하는 주요 버튼을 노출하지 마세요. 원본 사업으로 이동할 때도 읽기 전용 상태로 열어주세요.

### 화면그룹 I. 계정·권한 관리자

역할·관할 부여, 계정 관리, 결재정책 설정은 일반 정책 업무와 분리합니다.

기본 메뉴:

- 관리자 계정
- 역할·관할
- 결재정책
- 접근 이력

이 화면그룹에서는 “새 정책 수립”을 절대 노출하지 마세요.

## 여러 역할을 가진 사용자의 진입과 전환

### 로그인 순서

1. 행정 SSO 또는 공동인증서로 먼저 인증합니다.
2. 인증 결과로 접근 가능한 업무영역을 조회합니다.
3. 업무영역이 하나면 해당 시작 화면으로 바로 이동합니다.
4. 업무영역이 여러 개면 업무영역 선택 화면을 보여줍니다.

### 업무영역 선택 화면

카드마다 다음을 보여주세요.

- 정책 업무
- 사업·명부 업무
- 결재 업무
- 지급 실행 업무
- 발행 업무
- 정산 업무
- 감사 업무
- 계정·권한 관리

각 카드에는 역할 코드가 아니라 다음 정보가 필요합니다.

- 이 영역에서 하는 일 한 문장
- 현재 처리할 업무 건수
- 기한이 지난 업무 건수
- 마지막으로 처리한 사업

### 업무영역 전환

- 화면 상단에 현재 업무영역과 관할을 한글로 표시합니다.
- 여러 역할이 있는 사람에게만 “업무영역 전환”을 제공합니다.
- 전환하면 왼쪽 메뉴, 첫 화면, 가능한 주요 행동이 함께 바뀌어야 합니다.
- 역할을 전환해도 현재 사업을 이어볼 수 있으면 같은 사업 ID를 유지하되, 새 역할에서 허용된 읽기·편집 상태로 다시 엽니다.
- 권한이 없는 딥링크에 접근하면 빈 화면이나 오류 코드 대신 접근할 수 없는 이유와 권한 요청 경로를 보여줍니다.

## 한 사업을 중심으로 한 통합 업무 페이지

정책과 운영계획을 합친 결과물의 중심은 “바우처 사업 업무 상세”입니다.

이 페이지는 모든 정보를 한 편집폼으로 길게 나열하는 화면이 아닙니다. 한 사업의 전체 문맥과 다음 행동을 한 자리에서 보고, 책임이 다른 상세 작업으로 들어갔다가 같은 페이지로 돌아오는 업무 허브입니다.

### 1. 상단 사업 문맥

항상 고정해서 보여줄 정보:

- 정책명
- 가칭 또는 확인 필요 여부
- 사업연도
- 담당부서와 담당자
- 현재 업무상태
- 현재 결재상태
- 사업 회차
- 시행기간 또는 지급예정일
- 최근 변경일
- 처리기한
- 현재 사용자가 맡은 역할
- 현재 사용자가 해야 할 다음 행동

주요 상태를 “승인됨” 하나로 뭉치지 말고 다음을 구분해서 보여주세요.

- 정책 승인 여부
- 명부 확정 여부
- 지급 준비 여부
- 발행 준비 여부
- 지급 실행 결과

### 2. 정책과 운영 내용

- 사업 목적
- 행정·법적 근거
- 문서번호
- 근거문서와 첨부
- 대상 조건의 사람이 읽는 문장
- 1인 지급액
- 예상 대상 수
- 예상 지급 총액
- 사업 회차와 주요 일정
- 사용 지역·업종·기간
- 지급 형태
- 종료 시 잔액 처리 원칙

정책 담당자에게는 작성·보완 상태에서 편집을 허용하고, 다른 역할에는 읽기 전용으로 보여주세요.

### 3. 대상·명부 현황

- 대상 선정 방식
- 명부 원천
- 명부 기준일
- 현재 명부 버전
- 전체 건수
- 정상 건수
- 주의 건수
- 오류 건수
- 제외 건수
- 확정 대상 수
- 확정 지급 총액
- 명부 결재상태
- 마지막 업로드 담당자와 일시

오류가 있으면 “오류 37건”처럼 바로 보이게 하고 사업·명부 담당자에게만 “오류 처리” 행동을 제공합니다.

### 4. 예산·재원·발행 준비

- 정책 사업비 또는 재원 근거
- 지급 예정 총액

- 기존 발행물량 사용 가능 여부
- 신규 발행 검토 필요 여부
- 발행 준비상태
- 지급을 막는 미완료 항목

각 항목에는 담당자 이름, 역할, 부서, 시각과 결과가 필요합니다.

## 첫 PoC의 구체적인 바우처 업무 흐름

### 1. 정책 담당자 진입

정책 담당자는 로그인 후 “내 정책 업무”로 이동합니다. 기본 탭은 “내 처리 필요”이고, 보완 요청을 받은 업무와 기한 임박 업무를 먼저 봅니다.

목록에 우선 표시할 정보:

- 사업연도
- 정책명
- 담당부서
- 대상 요약
- 예상 대상 수
- 지급 예정 총액
- 현재 업무상태
- 결재상태
- 명부·지급 준비 요약
- 처리기한
- 다음 행동

정책 행을 누르면 바로 편집폼이 아니라 “바우처 사업 업무 상세”로 이동합니다.

### 2. 초안 작성

“새 바우처 사업”을 눌러 고정형 입력 화면을 엽니다.

입력 묶음:

1. 기본정보와 근거
2. 대상과 지급
3. 사용 조건
4. 운영 일정과 재원
5. 검토와 결재

화면 수를 다섯 장으로 고정하지 말고 한 페이지 안의 명확한 구역으로 구성하세요.

가능 행동:

- 임시저장
- 검토하기
- 결재 상신

“저장 및 다음”처럼 목적이 모호한 버튼은 사용하지 마세요.

### 3. 상신 전 검토

검토 결과는 오류와 주의를 구분합니다.

- 필수 근거 누락
- 첨부 누락
- 대상 조건과 사용 조건 충돌
- 지급액·예상 대상 수·총액 불일치
- 시행기간과 사용기간 충돌
- 재원 부족 가능성
- 발행 검토 필요

오류가 있으면 상신을 막고 수정 위치로 이동할 수 있게 합니다. 주의는 근거를 확인하고 상신할 수 있게 합니다.

### 4. 정책 결재

결재자는 내 결재함에서 해당 사업을 열어 다음을 비교합니다.

- 현재 제출안
- 이전 버전
- 근거와 첨부
- 대상 수와 총액
- 검증 결과

행동:

- 보완 요청
- 반려
- 승인

승인과 동시에 지급하거나 발행 서명하지 마세요.

### 5. 회차·명부 작업

정책 승인 후 사업·명부 담당자의 “내 명부·지급 준비 업무”에 새 업무가 나타납니다.

첫 PoC에서는 확정 명부 기반 직접 지급을 작업용 가정으로 사용합니다.

파일 처리 흐름:

1. 원본 파일 선택
2. 파일 형식과 필수 열 확인
3. 전체 행 검증
4. 정상·주의·오류·제외 분류
5. 오류 사유와 수정 방법 표시
6. 오류 목록 다운로드
7. 수정본 재업로드
8. 버전별 차이 확인
9. 확정 대상 수와 총액 확인
10. 명부 결재 상신

업로드 실패와 행 단위 오류를 구분하고, 전체를 다시 처음부터 하게 만들지 마세요.

### 6. 명부 결재

명부 결재자는 정책 조건과 명부 집계가 맞는지 확인합니다.

반드시 보여줄 것:

- 정책명과 버전
- 회차
- 명부 기준일과 버전
- 원본 파일
- 전체·정상·주의·오류·제외 건수
- 확정 대상 수
- 총 지급액
- 이전 명부 버전과 달라진 건

승인 결과는 지급 준비 상태를 만들 뿐 실제 지급을 시작하지 않습니다.

### 7. 지급 준비

지급 담당자의 목록에는 다음 준비조건이 모두 보입니다.

- 정책 승인 완료
- 명부 승인 완료
- 재원 확인
- 유통 단위 준비
- 발행 준비
- 지급예정일

준비되지 않은 항목이 있으면 실행 버튼을 비활성화하고, 무엇이 막혔고 어느 담당자가 해결해야 하는지 보여주세요.

### 8. 지급 실행

작업용 가정으로 지급 담당자가 별도 화면에서 수동으로 시작합니다.

실행 전 확인:

- 정책명
- 회차
- 명부 버전
- 대상 수
- 총액
- 중복 실행 여부
- 재원·발행 준비상태

재인증 후 지급을 시작하고, 진행 중에는 중복으로 다시 실행할 수 없게 하세요.

### 9. 진행·실패·완료

지급 결과를 다음처럼 나눕니다.

- 대기
- 검증 중
- 처리 중
- 완료
- 보류
- 실패

부분 실패 시 성공 건을 다시 지급하지 않고 실패 건만 재처리합니다. 실패 사유와 재처리 가능 여부를 사람이 이해할 수 있는 문장으로 보여주세요.

### 10. 발행사 인계

신규 발행이 필요할 때만 발행사 콘솔의 “발행 필요 정책”에 나타납니다.

발행 담당자는 다음을 읽기 전용으로 확인합니다.

- 정책명
- 담당 지자체와 부서
- 정책 승인일
- 시행기간과 지급예정일
- 대상 수와 지급 총액
- 명부 준비상태
- 기존 발행물량
- 신규 발행 필요량

발행 계획 작성, 승인, 최종 서명, 발행 완료를 분리합니다. 결과는 원래 바우처 사업 업무 상세로 다시 연결합니다.

## 상태와 가능한 행동

### 정책 상태

| 상태 | 담당자가 이해할 의미 | 정책 담당자 행동 | 결재자 행동 |
|---|---|---|---|
| 작성 중 | 초안을 만드는 중 | 임시저장, 검토, 삭제 | 없음 |
| 상신 전 검토 | 누락과 충돌을 확인하는 중 | 수정, 결재 상신 | 없음 |
| 결재 대기 | 결재자가 검토 중 | 진행 보기, 허용 시 상신 취소 | 보완, 반려, 승인 |
| 보완 요청 | 지정된 내용을 고쳐야 함 | 보완 작성, 재상신 | 보완 결과 대기 |
| 반려 | 현재 안으로 진행하지 않음 | 사유 확인, 개정안 작성, 종결 | 처리 결과 조회 |
| 정책 승인 | 정책 판단이 끝남 | 승인 결과 조회 | 처리 결과 조회 |

### 명부 상태

| 상태 | 의미 | 사업·명부 담당자 행동 | 명부 결재자 행동 |
|---|---|---|---|

## 반드시 추가할 화면

### 1. 업무영역 선택

여러 역할을 가진 사용자가 인증 후 정책, 명부, 결재, 지급, 발행, 정산, 감사 중 현재 할 일을 선택하는 화면입니다.

### 2. 바우처 사업 업무 상세

정책과 운영계획을 같은 사업 문맥으로 통합하는 중심 화면입니다. 역할에 따라 편집 가능 영역과 주요 행동이 달라집니다.

### 3. 정책 결재 상세

정책 근거, 첨부, 대상 수, 총액, 변경사항을 보고 보완·반려·승인을 판단합니다.

### 4. 명부 작업 목록

사업·명부 담당자가 자신에게 배정된 회차와 명부 업무를 찾는 시작 화면입니다.

### 5. 명부 업로드·검증

파일 업로드, 전체 검증, 오류 목록, 재업로드, 버전 비교, 확정 총액과 상신을 처리합니다.

### 6. 지급 업무 목록

지급 담당자가 실행 대기, 차단, 처리 중, 일부 실패 업무를 우선순위에 따라 찾습니다.

### 7. 지급 준비·실행 상세

정책·명부·재원·발행 준비를 확인하고 재인증 후 지급을 시작합니다.

### 8. 지급 진행·결과

대기·처리·완료·보류·실패 건과 실패 건 재처리 결과를 보여줍니다.

### 9. 발행 최종 서명

발행 승인과 분리된 최종 서명자의 재인증·다중서명 화면입니다.

## 첫 생성에서 우선할 화면

모든 기존 화면을 다시 생성하려 하지 말고 다음 순서로 우선 생성해주세요.

1. 인증 후 업무영역 선택
2. 정책 담당자의 내 정책 업무
3. 바우처 사업 업무 상세
4. 바우처 정책 작성·보완
5. 결재자의 내 결재함
6. 정책 결재 상세
7. 사업·명부 담당자의 내 업무
8. 명부 업로드·검증
9. 명부 결재 상세
10. 지급 담당자의 실행 대기
11. 지급 준비·실행
12. 지급 진행·일부 실패·완료
13. 발행 필요 정책
14. 발행 계획 승인
15. 발행 최종 서명

정산, 감사, 계정·권한, 처분 화면은 기존 Spec에 보존하되 첫 생성 묶음에서는 역할별 메뉴 연결만 정확히 정리해도 됩니다.

## 작업용 가정과 미확정 항목

### Spec 생성에 사용할 작업용 가정

다음은 흐름을 연결하기 위한 임시 가정이며 Spec 안에서도 작업용 시나리오라고 표시합니다.

1. 첫 정책은 “청년 지역화폐 지급 바우처”라는 가칭을 사용한다.
2. 첫 지급 방식은 확정 명부 기반 직접 지급이다.
3. 정책 승인과 명부 승인은 서로 다른 단계다.
4. 명부 승인 뒤에도 지급은 자동 시작되지 않는다.
5. 지급 실행 담당자가 준비상태를 확인하고 별도로 지급을 시작한다.
6. 발행 준비가 완료되지 않으면 지급 시작을 차단한다.
7. 신규 발행이 필요할 때만 발행사 콘솔로 인계한다.

### 확정값으로 넣지 말아야 할 것

- 청년과 청소년 중 정확한 정책명
- 실제 대상 연령
- 거주기간과 제외조건
- 지급액
- 회차와 신청기간
- 신청형인지 행정 명부형인지에 대한 최종 사업 결정
- 실제 명부 원천 시스템
- 실제 지급예정일
- Reserve 10%
- D+2
- 운영 Hold
- 환불·소각의 최종 모델

이 값들은 현실처럼 임의로 확정하지 말고 “정책 시나리오 확인 필요” 표식을 사용하거나 예시 데이터임을 명확히 표시하세요.

## 접근성과 사용성 요구

1. 모든 메뉴와 버튼은 키보드로 접근 가능해야 합니다.
2. 현재 초점이 분명하게 보여야 합니다.
3. 긴 표는 200% 확대에서도 주요 열과 행 행동을 이해할 수 있어야 합니다.
4. 오류는 색만 사용하지 말고 행 번호, 오류 사유, 수정 방법을 텍스트로 제공합니다.
5. 파일 업로드 진행, 검증 완료, 실패와 지급 상태 변경은 보조기기에 전달 가능한 상태 메시지를 사용합니다.
6. 아이콘만 있는 행 행동을 사용하지 말고 텍스트 라벨 또는 명확한 접근 가능한 이름을 제공합니다.
7. 오른쪽 긴 패널에 주요 버튼을 고정해 잘리게 만들지 마세요.
8. 승인·지급·서명 같은 위험 행동은 대상, 금액, 근거를 다시 확인하고 재인증할 수 있게 합니다.
9. 읽기 전용 값과 편집 가능한 값을 시각적으로 명확히 구분합니다.
10. 비활성화된 버튼 옆에는 왜 실행할 수 없는지와 해결 방법을 표시합니다.

## Spec 변환 workflow의 출력 지시

1. 기존 refined-production-spec.json 전체 구조와 관련 없는 화면을 보존한 완전한 JSON을 출력하세요.
2. 이 문서의 역할별 화면그룹을 사용자 권한, 내비게이션, 시작 화면과 화면 접근범위에 반영하세요.
3. 한 사용자가 여러 역할을 가질 수 있지만 현재 업무영역의 메뉴만 보이도록 역할 전환 구조를 추가하세요.
4. 기존 관리 콘솔과 발행사 콘솔의 권한 루트를 합치지 마세요.
5. 바우처 사업 업무 상세를 정책과 운영계획의 공통 문맥 화면으로 추가하세요.
6. 역할별로 같은 사업 화면을 열더라도 읽기·편집·승인·실행 행동이 다르게 보이게 하세요.
7. 정책 결재, 명부 결재, 지급 실행, 발행 승인, 최종 서명을 별도 상태와 상호작용으로 정의하세요.
8. 정책 목록에서 지급 결과까지 이어지는 종단 시나리오를 데모 흐름에 추가하세요.
9. 새 화면마다 기본, 로딩, 빈 상태, 오류, 권한 없음, 읽기 전용, 처리 완료 상태를 정의하세요.
10. 첫 PoC 생성 우선순위와 후순위 화면을 명시하세요.
11. 기존 화면 식별자는 의미가 유지되는 경우 보존하고, 새 화면에는 중복되지 않는 식별자를 부여하세요.
12. 기술 식별자는 Spec 내부 계약으로 보존하되 화면 문구에는 노출하지 마세요.

## 최종 수용 시나리오

다음 흐름을 역할을 바꿔가며 화면으로 설명할 수 있어야 합니다.

정책 담당자가 내 처리 필요 업무에서 바우처 사업을 찾는다.

→ 같은 사업 페이지에서 근거, 대상, 금액, 일정과 현재 준비상태를 이해한다.

→ 정책 초안을 작성하고 검토한 뒤 결재 상신한다.

→ 정책 결재자가 내 결재함에서 근거와 변경사항을 보고 보완 요청한다.

→ 정책 담당자가 보완해 재상신하고 결재자가 승인한다.

→ 사업·명부 담당자의 내 업무에 해당 사업의 명부 작업이 생성된다.

→ 명부 파일을 업로드하고 오류를 확인해 수정본을 재업로드한다.

→ 명부 결재자가 대상 수, 총액과 버전을 확인해 승인한다.

→ 지급 담당자의 실행 대기에 해당 사업이 나타난다.

→ 지급 담당자가 정책·명부·재원·발행 준비를 확인하고 재인증 후 지급을 시작한다.

→ 진행 중, 일부 실패, 실패 건 재처리와 완료 결과를 확인한다.

→ 신규 발행이 필요한 경우 발행사 콘솔에서 발행 계획을 작성한다.

→ 발행 승인자가 승인하고 최종 서명자가 별도로 서명한다.

→ 모든 담당자가 같은 바우처 사업 업무 상세에서 자신에게 허용된 범위로 정책, 명부, 지급과 발행 결과를 다시 확인한다.

## 최종 검수 체크리스트

- [ ] 로그인 전에 콘솔을 고르도록 강요하지 않는다.
- [ ] 역할이 하나면 해당 시작 화면으로 바로 이동한다.
- [ ] 역할이 여러 개면 업무영역을 선택하고 전환할 수 있다.
- [ ] 업무영역을 전환하면 왼쪽 메뉴와 주요 행동이 실제로 바뀐다.
- [ ] 정책 담당자 화면에 정산·거버넌스·발행 실행 메뉴가 기본 노출되지 않는다.
- [ ] 발행사 콘솔에 새 정책 수립 버튼이 없다.
- [ ] 계정·권한 화면에 새 정책 수립 버튼이 없다.
- [ ] 정책 목록이 데이터 분류표가 아니라 내 업무 목록으로 읽힌다.
- [ ] 정책과 운영계획이 한 사업 업무 상세에서 연결된다.
- [ ] 화면 통합이 권한 통합으로 구현되지 않는다.
- [ ] 정책 담당자가 명부를 최종 승인하거나 지급을 실행할 수 없다.
- [ ] 결재자가 결재 화면에서 원안을 직접 수정할 수 없다.
- [ ] 지급 담당자가 정책이나 확정 명부를 수정할 수 없다.
- [ ] 정책 승인과 명부 승인이 구분된다.
- [ ] 명부 승인과 지급 시작이 구분된다.
- [ ] 발행 승인과 최종 서명이 구분된다.
- [ ] 명부 업로드 오류와 행 단위 오류를 구분한다.
- [ ] 부분 실패 시 실패 대상만 재처리한다.
- [ ] 상태와 가능한 행동이 모순되지 않는다.
- [ ] 내부 코드보다 사람이 읽는 업무명이 먼저 보인다.
- [ ] 정확한 정책명·연령·금액·회차를 확인 전 확정하지 않는다.
- [ ] 첫 PoC에서 바우처 한 건의 종단 흐름을 우선 생성한다.

## 화면 증거의 한계

이번 판단은 기존 Spec과 현재 저장된 정적 Stitch 화면을 함께 본 결과입니다. 정적 이미지에서는 실제 역할 전환, 키보드 이동, 초점 순서, 업로드 검증, 결재 상태 변경, 지급 재처리와 권한 차단이 동작하는지 확인할 수 없습니다. 업데이트 Spec으로 화면을 다시 생성한 뒤에는 역할별로 로그인하여 위 최종 수용 시나리오를 실제 클릭 흐름으로 다시 검증해야 합니다.
